🔐 Security Architecture
▸ JWT Auth — HttpOnly cookies, @token_required decorator on every web route
▸ RBAC — @permission_required + ROLE_PERMISSIONS map; GET-bypass closed (Phase 0)
▸ RVM Auth — X-API-Key BCrypt-hashed server-side; rotate-api-key endpoint
▸ QR HMAC — HMAC-SHA256(org_secret, display_id)[:6]; Fernet-encrypted per org
▸ Multi-tenant isolation — org-scoped queries; cross-tenant leakage prevented at ORM layer
▸ CSRF — double-submit cookie; AUTH_CSRF_DISABLED=true in local dev only
▸ Rate limiting — Flask-Limiter on auth endpoints
▸ Secret hygiene — tools/tests/test_secret_hygiene.py; 45 tests run in CI
⚡ Resiliency & Failure Modes
1. RVM Network Loss ⚠️ Open Gap
Current firmware makes direct API calls per event — no SQLite offline buffer implemented yet. Network loss during a session will fail the deposit. Mitigation needed: add offline queue + replay to main.py before production Pi deployment.
2. Redis Unavailable
cache.py catches redis.exceptions.ConnectionError → falls back to direct Supabase query · zero data loss · ~200–500ms latency spike only
3. Supabase Pool Exhaustion (Most Severe)
Session pooler :6543 multiplexes connections · Gunicorn sync workers each hold 1 conn · RVM SQLite absorbs outage window · /api/health for readiness probing
