# RPI API Contract — Post-Phase 4A Hardening

> Committed deliverable for the hardware team. This document fully specifies
> the post-hardening `/api/rpi/*` interface that every RVM must implement
> against on first boot.

## Authentication

Every request to `/api/rpi/*` **must** include:

| Header | Value | Notes |
| --- | --- | --- |
| `X-API-Key` | Plaintext API key provisioned via `POST /api/web/machines/<id>/rotate-api-key` | One-time display; stored BCrypt-hashed server-side |
| `Content-Type` | `application/json` | Required for all POST bodies |

The `machineUuid` field in the request body identifies the RVM. The server
looks up the RVM by `machine_uuid`, then constant-time compares the
`X-API-Key` header against the stored BCrypt hash.

### Error responses

| Code | HTTP | When |
| --- | --- | --- |
| `RPI_AUTH_INVALID` | 401 | Missing/invalid `X-API-Key` or `machineUuid` |
| `RPI_MACHINE_UNKNOWN` | 404 | `machineUuid` not found in database |
| `QR_HMAC_INVALID` | 401 | HMAC suffix on QR payload fails verification |

All error responses follow the envelope:

```json
{
  "success": false,
  "error": {
    "code": "<ERROR_CODE>",
    "message": "<human-readable>"
  }
}
```

## HMAC-Signed QR Payload Format

User QR codes encode a **signed payload**:

```
<display_id>.<hmac_suffix>
```

| Part | Example | Description |
| --- | --- | --- |
| `display_id` | `USER-AU-001` | User's `display_id` from the database |
| `.` | `.` | Literal dot separator (split on rightmost `.`) |
| `hmac_suffix` | `a3f1b2` | First 6 hex chars of `HMAC-SHA256(per_org_secret, display_id)` |

**Full example**: `USER-AU-001.a3f1b2`

The per-org HMAC secret is provisioned server-side and stored encrypted.
The RVM never sees the secret — QR codes are pre-generated by the server
or a trusted admin tool.

### Validation flow (server-side)

1. Split `qrPayload` on rightmost `.` → `display_id`, `hmac_suffix`
2. Compute `HMAC-SHA256(org_secret, display_id)[:6]` (lowercase hex)
3. Constant-time compare via `hmac.compare_digest`
4. **Mismatch** → HTTP 401 `QR_HMAC_INVALID` (NO database user lookup)
5. **Match** → proceed to `User.query.filter_by(display_id=display_id)`

## API Key Provisioning Flow

1. **Superadmin** calls `POST /api/web/machines/<id>/rotate-api-key`
2. Server generates a 32-byte random key (`secrets.token_urlsafe(32)`)
3. Server stores BCrypt hash in `rvms.api_key_hash`
4. Server returns the plaintext key **exactly once** in the response
5. Operator copies the key to the RVM's configuration file
6. Subsequent requests from the RVM include the key in `X-API-Key`

**Key rotation**: call the same endpoint again. Previous key is immediately invalidated.

---

## Per-Org HMAC Secret Provisioning

- Auto-provisioned by the seeder for development
- For production: a superadmin rotates via a future admin UI or CLI command
- Secret is stored Fernet-encrypted in `organizations.qr_hmac_secret_enc`
- Encryption key is derived from `SECRET_KEY` via HKDF (no separate env var)
- **Rotation**: write new ciphertext → all previously issued QR codes become invalid

## Endpoints

### POST /api/rpi/machine/identify

Identify and authenticate an RVM.

**Request**:
```json
{
  "machineUuid": "RVM-EPTU-SEED-001"
}
```

**Response** (200):
```json
{
  "success": true,
  "machine": {
    "id": 1,
    "name": "Seed RVM",
    "locationName": "Main Campus",
    "isOnline": true,
    "isCapacityFull": false,
    "organizationId": 1
  }
}
```

### POST /api/rpi/session/{session_id}/deposit

Deposit a single item during an active session.

**Request**:
```json
{
  "detectedClass": "PET Bottle - Small",
  "confidenceScore": 95.50,
  "pointsAwarded": 5,
  "status": "Accepted"
}
```

**Response** (201):
```json
{
  "success": true,
  "item": {
    "id": 1,
    "detectedClass": "PET Bottle - Small",
    "confidenceScore": 95.5,
    "pointsAwarded": 5,
    "status": "Accepted"
  },
  "sessionTotals": {
    "itemCount": 1,
    "totalPointsEarned": 5
  }
}
```

> **Note**: `status` must be `"Accepted"` or `"Rejected"`. Rejected items award 0 points.

---

### POST /api/rpi/session/{session_id}/end

End an active session and credit points to the wallet.

**Request**:
```json
{
  "status": "completed"
}
```

| `status` value | Wallet credited? |
| --- | --- |
| `completed` | Yes (if points > 0) |
| `timed_out` | No |
| `error` | No |

**Response** (200):
```json
{
  "success": true,
  "session": {
    "id": 1,
    "status": "completed",
    "itemCount": 3,
    "totalPointsEarned": 15,
    "startTime": "2026-05-27T08:00:00+00:00",
    "endTime": "2026-05-27T08:05:00+00:00"
  }
}
```

### GET /api/rpi/config/points/{org_id}

Get point-per-item configuration. Requires `X-API-Key` + `machineUuid` in the request body (send as POST-style GET with body, or include `machineUuid` as a query param — the decorator reads the body).

> **Note**: This is a GET endpoint. The `machineUuid` must still be resolvable for `@rpi_auth_required`. Send it in a JSON body.

**Response** (200):
```json
{
  "success": true,
  "config": {
    "smallWithLabel": 5,
    "smallNoLabel": 3,
    "mediumWithLabel": 8,
    "mediumNoLabel": 5,
    "largeWithLabel": 10,
    "largeNoLabel": 7
  }
}
```

## Signed QR Generator — Python Example

```python
#!/usr/bin/env python3
"""Generate an HMAC-signed QR payload for an EcoPoints user."""
import hashlib
import hmac
import sys


def make_qr_payload(secret: bytes, display_id: str) -> str:
    """Return '<display_id>.<6-char HMAC suffix>'."""
    digest = hmac.new(secret, display_id.encode(), hashlib.sha256).hexdigest()
    return f"{display_id}.{digest[:6]}"


if __name__ == "__main__":
    # Usage: python qr_sign.py <hex_secret> <display_id>
    secret = bytes.fromhex(sys.argv[1])
    did = sys.argv[2]
    print(make_qr_payload(secret, did))
```

**Usage**: `python qr_sign.py deadbeef... USER-EPTU-001`
